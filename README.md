# Payment Gateway ‚Äì Design Document

## Overview

This project implements a payment gateway that allows merchants to process card payments and
retrieve payment details. The gateway validates incoming requests, forwards them to a simulated
acquiring bank (implemented with Mountebank), stores the outcome, and returns the appropriate
responses. The solution is built with **Java 17**, **Spring Boot 3**, and uses **Gradle** as the
build tool. **JUnit 5** with **Mockito** is used for testing.


## Key Design Considerations

### 1. Architecture

- **Layered Design**: Controller ‚Üí Service ‚Üí Repository ‚Üí External Client.
  - **Controller** handles HTTP requests, input validation, and response formatting.
  - **Service** contains business logic, orchestrates calls to the bank client, and manages
    transactions.
  - **Repository** abstracts storage (in‚Äëmemory `ConcurrentHashMap` for this assessment).
  - **Bank Client** isolates communication with the external bank simulator, making it easy to
    replace with a real acquirer later.

### 2. Validation

- **Bean Validation (Jakarta Validation)** is used on the `PostPaymentRequest` to enforce basic
  field constraints at the controller layer. This rejects malformed requests immediately (HTTP 400)
  without calling the service.
- **Required Fields**: To support Bean Validation‚Äôs `@NotNull`, primitive types (`int`) were changed
  to their wrapper types (`Integer`) for `expiryMonth`, `expiryYear`, and `amount`. This allows the
  validation framework to detect missing fields.
- **Custom Expiry Date Validation**: Because the expiry check depends on both month and year, it is
  performed in the service layer. If the combination is in the past, an `EventProcessingException`
  is thrown, resulting in a `400 Bad Request` (Rejected).
- **Currency Support**: The assessment requires validation against no more than 3 currency codes. We
  restrict to `USD`, `EUR`, and `GBP` (the most common e‚Äëcommerce currencies). This can easily be
  extended by modifying the `@Pattern` regex.

### 3. Idempotency

- **Goal**: Ensure that a merchant can safely retry a payment request without creating duplicate
  payments.
- **Implementation**:
  - The client provides an optional `Idempotency-Key` header.
  - A dedicated in‚Äëmemory cache (`ConcurrentHashMap<String, PostPaymentResponse>`) stores the
    response for each successfully processed payment key.
  - If the same key is received for a subsequent request, the previously stored response is returned
    immediately without calling the bank again.
  - Idempotency keys are only effective for **successful payments** (those that resulted in a
    stored payment).
  - The cache is **ephemeral**, it is cleared when the application restarts. For production, a
    persistent store (e.g., Redis) would be recommended.
- **Assumptions**:
  - Idempotency keys are generated by the merchant and must be unique per request.
  - The gateway does not validate the format of the key.
  - If a request with an existing key fails (e.g., validation error or bank unavailable), the cache
    is not updated, allowing the merchant to retry with the same key.

### 4. PCI Compliance

- **No storage of full PAN or CVV**: Only the **last four digits** of the card number are stored in
  the repository. The full card number is never persisted; it is only used for the bank call and
  then discarded.
- **Logging**: The `toString()` method of `PostPaymentRequest` masks the card number (showing only
  last four digits) and CVV (`***`). This ensures sensitive data does not accidentally appear in
  logs.

### 5. Bank Simulator Integration

- The bank simulator is provided as a Mountebank container (`docker-compose up`). It listens on
  `http://localhost:8080` and implements the following logic (as defined in `bank_simulator.ejs`):
  - Missing required fields ‚Üí HTTP 400
  - Card number ends in odd digit (1,3,5,7,9) ‚Üí 200 with `authorized: true` and a random
    `authorization_code`
  - Card number ends in even digit (2,4,6,8) ‚Üí 200 with `authorized: false` and empty
    `authorization_code`
  - Card number ends in 0 ‚Üí 503 Service Unavailable
- The `BankSimulatorClient` uses `RestTemplate` to call the simulator. It catches `503` responses
  and re‚Äëthrows a custom `BankUnavailableException`, which the global handler maps to a `503`
  response to the merchant.
- **No wire mocking in integration tests**: The tests rely on the real Mountebank container,
  ensuring full end‚Äëto‚Äëend testing with the actual simulated bank.

### 6. Storage

- **In‚Äëmemory repository** (`ConcurrentHashMap`) satisfies the requirement ‚Äútest double repository‚Äù.
- No persistence is implemented; the repository is cleared before each integration test.
- The repository stores `PostPaymentResponse` objects, which already contain only the masked card
  data.
- **Idempotency cache** is separate from the main payment repository, allowing fast lookup without
  mixing concerns.

### 7. Error Handling

- A `@ControllerAdvice` (`CommonExceptionHandler`) centralises error responses.
- It maps:
  - `MethodArgumentNotValidException` ‚Üí 400 (validation errors)
  - `EventProcessingException` (custom validation failure, e.g., expired card) ‚Üí 400
  - `PaymentNotFoundException` ‚Üí 404
  - `BankUnavailableException` ‚Üí 503
  - Any unhandled exception ‚Üí 500
- All error responses use the provided `ErrorResponse` class with a user‚Äëfriendly message.

### 8. Testing Strategy

- **Unit Tests**: Service layer and client layer are tested with **JUnit 5** and **Mockito**. This
  verifies business logic in isolation, including idempotency key behaviour.
- **Integration Tests**: The `PaymentControllerTest` uses `@SpringBootTest` with a random
  port and the real Mountebank container. It exercises the full stack, including HTTP serialisation,
  validation, and the bank simulator.

### 9. API Design

- **POST /payments** ‚Äì Processes a payment. Accepts an optional `Idempotency-Key` header.
  Returns `202 Accepted` (HTTP 202) with the payment details.
- **GET /payments/{id}** ‚Äì Retrieves a previously processed payment. Returns `200 OK` with the
  stored details (including masked card number) or `404 Not Found`.

### 10. Model Corrections

The provided model classes required several adjustments to meet the functional and validation
requirements:

- **`PostPaymentRequest`**:
  - The original field `cardNumberLastFour` (type `int`) was replaced with `String cardNumber` (with
    `@JsonProperty("card_number")`) to accept the full card number as required.
  - `expiryMonth`, `expiryYear`, and `amount` were changed from primitive `int` to `Integer` to
    allow `@NotNull` validation. This ensures the framework can detect missing fields and return a
    proper validation error.

- **`GetPaymentResponse` and `PostPaymentResponse`**:
  - `cardNumberLastFour` and `cvv` were changed from primitive `int` to `String` because CVV can be
    `012`. If stored as an int, it becomes `12`. Card number last four can be `005`. If stored as an
    int, it becomes `5`.

- **`CommonExceptionHandler`**
  - Was extended to handles custom exceptions

All other provided classes (`ErrorResponse`, `PaymentStatus`, `EventProcessingException`) are used
as given.


## Assumptions

1. **Bank Simulator Always Available**: The test expects the Mountebank container to be started
   manually (`docker-compose up`) before running integration tests. In a CI environment, this would
   be automated.
2. **Currency List**: Only three currencies (`USD`, `EUR`, `GBP`) are validated; the merchant‚Äôs
   request must use one of them. This meets the requirement of ‚Äúno more than 3 currency codes‚Äù.
3. **Minor Currency Units**: The `amount` field is expected in the smallest unit of the currency (
   e.g., pence for GBP). This is validated as a positive integer.
4. **Card Number Validation**: The regex `\d{14,19}` ensures only numeric characters and the correct
   length. Luhn checks are not required.
5. **CVV Validation**: Allowed 3‚Äì4 digits. No storage or validation beyond format.
6. **Expiry Date**: The combined month/year must be in the future at the time of processing. The
   check uses the system‚Äôs current date.
7. **Payment ID Format**: A UUID (version 4) is generated for each successful payment.
8. **Idempotency Key**: The key is optional. If provided, it must be unique per payment attempt. The
   gateway does not generate keys; the merchant is responsible for ensuring uniqueness.
9. **Idempotency Cache Lifetime**: The cache lives only for the duration of the application
   instance. Restarting the application clears the cache, which may allow duplicate payments if the
   same key is reused across restarts. In a production environment, a persistent cache (e.g., Redis)
   would be used.



## How to Run

### Prerequisites

- Java 17 or later
- Docker and Docker Compose (for the bank simulator)
- Gradle (the project includes a Gradle wrapper, so you don't need to install it globally)

### Step 1: Start the Bank Simulator

From the project root (where `docker-compose.yml` is located), run:

```bash
docker-compose up
```

The simulator will be available at `http://localhost:8080`.

### Step 2: Build and Run the Payment Gateway

```bash
./gradlew clean build
./gradlew bootRun
```

The gateway will start on `http://localhost:8090`.

### Step 3: Test the API

**Process a payment (authorized):**

```bash
curl -X POST http://localhost:8090/payments \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: unique-key-123" \
  -d '{
    "card_number": "2222405343248877",
    "expiry_month": 4,
    "expiry_year": 2027,
    "currency": "GBP",
    "amount": 100,
    "cvv": 123
  }'
```

**Retrieve a payment:**

```bash
curl http://localhost:8090/payments/{id}
```

## Running Tests

- **Unit tests** (service layer and client) can be run with:
  ```bash
  ./gradlew test
  ```
- **Integration tests** require the bank simulator to be running (see Step 1). Then execute:
  ```bash
  ./gradlew check
  ```

The tests are written with **JUnit 5** and **Mockito** and cover:

- Validation logic
- Idempotency key handling
- Bank client error handling
- End‚Äëto‚Äëend flow with the Mountebank simulator


## Future Enhancements

- Replace the in‚Äëmemory repository with a real database (e.g., PostgreSQL) for persistence.
- Use a persistent store (e.g., Redis) for idempotency cache to survive application restarts.
- Add idempotency key expiration to avoid unbounded cache growth.
- Integrate with an actual acquiring bank via their API.
- Add monitoring (e.g., Micrometer) and health checks.
- Implement Luhn check for card numbers.
- Add more comprehensive logging with structured data.
- Use asynchronous processing for payments to improve throughput.
